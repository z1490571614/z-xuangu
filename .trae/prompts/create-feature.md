# Prompt：创建功能规格 (Create Feature)

## 任务目标
帮用户把模糊的功能想法转化为包含高质量验收标准（AC）的需求文档，为后续技术设计和 TDD 开发提供依据。

## 必读上下文
| 文件 | 目的 |
|------|------|
| `specs/PROJECT-CONTEXT.md` | 了解项目全局 |
| `specs/产品概述.md` | 确保功能与项目全局一致 |
| `project-status.json` | 了解当前阶段 |

## 执行步骤
1. 通过自然对话挖掘需求（5 个维度：场景与用户、核心流程、边界与异常、业务规则、范围）
2. 每次只问 1-2 个问题，给示例降低回答难度
3. AC 从对话中自然生长——用户描述的每个流程/异常/规则都可能是一条 AC
4. 信息足够后进行总结：核心流程 + AC 列表 + 范围界定
5. 用户确认后生成 `specs/features/{功能名}.md`
6. 更新 `project-status.json` 中的 features 数组

## 文件修改边界
**会创建/修改**：`specs/features/{功能名}.md`、`project-status.json`
**不会修改**：任何代码文件、现有 specs 文档

## 输出格式
`specs/features/{功能名}.md`（使用 feature-spec-template.md）
AC 格式：`AC-001: Given {前置条件}, When {触发动作}, Then {预期结果}`

## 验证要求
- [ ] AC 覆盖正常、边界、异常三类场景
- [ ] 每条 AC 可观测、可测试
- [ ] 范围明确（做什么/不做什么）
- [ ] 全程不涉及技术实现细节

## 禁止行为
- 不讨论技术实现（API、数据库、代码）
- 不猜测用户没说的需求
- 不用技术术语写 AC
