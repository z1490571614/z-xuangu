# Prompt：Bug 修复 (Fix Bug)

## 任务目标
诊断并修复代码行为与需求/设计不一致的问题，确保不再复发。

## 必读上下文
| 文件 | 目的 |
|------|------|
| `specs/PROJECT-CONTEXT.md` | 项目全局认知 |
| 相关 feature 文档 | 了解期望行为 |
| 相关代码文件 | 定位问题 |

## 执行步骤

### 第 0 步：判断是不是 Bug
- **是 Bug**：代码行为与需求/设计/AC 不一致
- **不是 Bug**：需求变更（走 feature-evolution 流程）

### 简单问题 → 快速修复
1. 直接定位并修复
2. 跑相关测试
3. 告知用户

### 复杂问题 → 完整排查
1. **复现**：收集信息（位置、操作、期望 vs 实际、日志）
   - 未复现不动代码（铁律）
2. **定位根因**：模块 → 组件 → 函数，找逻辑分叉点
3. **修复**：最小改动，复用已有模式
4. **验证**：根据 Bug 类型选择方式（单元测试/Playwright/数据库查询）
5. **报告**：生成 `docs/BUG修复文档/YYYYMMDD-HHMM-问题简述.md`

## 文件修改边界
**会修改**：相关代码文件、测试文件
**不会修改**：无关模块、现有功能

## 输出格式
修复报告（复杂问题）包含：基本信息、问题、根因、修复、测试、遗留问题

## 验证要求
- [ ] 复杂问题已复现
- [ ] 修复后有验证（测试或手动）
- [ ] 已有测试全部通过（无回归）

## 禁止行为
- 未复现不动代码（复杂问题）
- 不把需求变更当 Bug 修
- 不顺手重构不相关代码
- 不删除测试让代码通过
