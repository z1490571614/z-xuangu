# 09 - Bug 修复 (Bug Fix)

## Purpose
定义从问题报告到修复完成的闭环流程。关键区分：Bug 是"代码行为与需求/设计不符"，不是需求变更。

## When to use
- 用户报告功能异常、不符合预期
- 代码行为与 AC 描述不一致
- **不适用**：需求变更（走 feature-evolution）、调整文案/样式（走功能迭代）

## Required inputs
- 用户的问题描述
- `specs/PROJECT-CONTEXT.md`（如果存在）
- 相关 feature 文档（搜索匹配）
- 相关代码文件

## Workflow

### 角色：擅长排查问题的开发者

### 第 0 步：判断是不是 Bug
动手前先判断：
- **是 Bug**：代码行为与需求/设计/AC 不一致。例："登录应跳首页，实际跳 404"
- **不是 Bug**：要新行为、改文案、加功能 → 建议走对应开发流程

### 问题分级

#### 简单问题 → 快速修复
- 一眼能看出问题在哪
- 改动范围很小（几行代码）
- 做法：定位 → 修复 → 跑测试 → 告知用户
- 不需要完整复现流程和修复报告

#### 复杂问题 → 完整排查
- 原因不明显
- 涉及多模块
- 逻辑复杂或影响面大

### 完整修复流程

**Step 1：收集信息，复现问题**
- 未复现不动代码（铁律）
- 收集：发生位置、操作步骤、期望 vs 实际、日志/截图
- 信息不够就追问，每次 1-2 个关键问题
- 复现成功后才进入下一步

**Step 2：定位根因**
- 从现象缩小范围：模块 → 组件/服务 → 函数
- 对照"期望行为 vs 实际行为"找逻辑分叉点
- 优先检查最近改动、复用模块、公共工具

**Step 3：修复**
- 最小改动原则
- 复用项目已有模式和工具
- 多方案时列出分析并推荐
- 顺手重构不相关代码

**Step 4：验证**
根据 Bug 类型选择验证方式：
- 逻辑/数据类：单元测试/集成测试
- UI/交互类：Playwright 浏览器验证
- 数据类：数据库查询验证
- 修复后跑全量测试防回归

**Step 5：生成报告**
- 参考 `assets/bug-report-template.md`
- 保存到 `docs/BUG修复文档/YYYYMMDD-HHMM-问题简述.md`
- 简单问题不需要报告

## Output format

Bug 修复报告包含：
- 基本信息（日期、模块、严重级别）
- 问题描述（期望 vs 实际、复现方式）
- 根因分析
- 修复内容（改了什么、为什么）
- 验证结果
- 遗留问题（如有）

## Quality checklist
- [ ] 确认问题是 Bug 而非需求变更
- [ ] 复杂问题已复现
- [ ] 修复后有验证（测试或手动）
- [ ] 复杂问题有修复报告
- [ ] 代码行为与文档一致（不一致则同步文档）

## Guardrails
- 未复现不动代码（复杂问题）
- 最小改动——不改不相关代码
- 先判断是不是 Bug
- 不删除测试让代码通过

## Common mistakes to avoid
- 把需求变更当 Bug 修复 → 走错流程，文档和代码对不上
- 看报错信息就猜原因动手改 → 需要找到根因
- 修复了现象没修复根因 → Bug 会换种方式再出现
