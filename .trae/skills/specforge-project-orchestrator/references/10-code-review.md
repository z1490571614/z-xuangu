# 10 - 代码审查 (Code Review)

## Purpose
对代码进行结构化审查，检查是否符合项目规范、技术方案、验收标准。支持两种场景：审查他人代码和审查 AI 生成的代码。

## When to use
- 用户说"帮我看看这段代码"、"审查一下"
- 用户需要对 AI 生成的代码把关
- 功能开发完成后，发 PR 前
- 重构后的代码审查

## Required inputs
- 要审查的代码文件
- 对应的 feature 文档（需求、技术方案、任务规划，如有）
- `specs/开发规范.md`
- `specs/PROJECT-CONTEXT.md`

## Workflow

### 角色：代码审查者

### 审查维度

**正确性**：
- 代码行为是否符合 AC 和技术方案描述？
- 边界情况和异常处理是否覆盖？
- 逻辑是否有明显错误或遗漏？

**规范性**：
- 命名是否符合开发规范？
- 代码风格是否一致？
- 类型使用是否正确（禁止 `any` 等规则）？

**设计**：
- 是否遵循已有的设计模式？
- 是否引入了不必要的耦合？
- 是否复用了已有的工具和抽象？

**安全**：
- 输入校验是否充分？
- 是否有注入风险（SQL、XSS）？
- 密钥/敏感信息是否暴露？

**测试**：
- 测试是否覆盖核心路径？
- 测试是否真实可信（不是"为了覆盖率而写"）？
- 边界和异常场景是否有测试？

**文档一致性**：
- 代码是否与 spec 文档一致？
- 不一致的地方是否需要更新文档？

### 审查输出

**轻度审查**（快速反馈）：
- 发现几处问题 → 给出位置、问题、建议

**全面审查**（正式 PR）：
- 参考 `assets/code-review-template.md`
- 输出结构化审查报告

## Output format

审查报告包含：
- 审查概要（范围、总体评价）
- 严重问题（必须修）
- 建议（可选改）
- 正面反馈（做得好的地方）
- 测试建议（如适用）

## Quality checklist
- [ ] 检查了正确性、规范性、设计、安全四个维度
- [ ] 每个问题有明确位置和修复建议
- [ ] 区分了"必须修"和"建议改"
- [ ] 检查了代码与文档的一致性

## Guardrails
- 不给"看起来没问题"的审查——至少确认了每个维度
- 不把个人偏好当问题——"我不喜欢这种写法"不是审查问题
- 不在代码审查中提出新的功能需求
- 不审查与当前任务无关的代码

## Common mistakes to avoid
- 只关注代码风格忽略逻辑正确性
- 看到不符合自己习惯的代码就提问题——区分"风格"和"问题"
- 不提建议只提问题——好的审查给出方向性建议
