# 03 - 项目接入 (Project Intake)

## Purpose
将用户的模糊想法转化为标准化的项目描述。这是项目启动的第一站——用户在说"我想做一个XX"时触发。

## When to use
- 用户说"我有一个想法"、"我想做个XX项目"、"帮我起个项目"
- `specs/` 目录为空或不存在
- `project-status.json` 不存在

## Required inputs
- 用户的初始想法描述
- 无需任何前置文档

## Workflow

### 1. 倾听与分析
接收用户的初始描述。不要急于输出模板或结构化内容。像感兴趣的合伙人一样倾听。

### 2. 引导式多维提问
通过自然对话帮用户想清楚：

**WHY — 为什么要做？**
- 解决了什么痛点？
- 是为了自己用、给团队用、还是商业化？

**WHO — 谁在用？**
- 核心用户是谁？是一个人还是很多人？

**WHAT — 做什么？**
- 核心功能是什么？完整愿景是什么？
- 有什么你觉得"必须要有"的东西？

**HOW — 有什么特殊要求？**
- 技术偏好？离线/在线？移动端/桌面端？
- 成本、时间、团队规模有什么约束？

### 3. 迭代澄清
每次只问 1-2 个最关键的问题。根据回答继续追问或转向其他维度。不给选项列表轰炸用户。

### 4. 最终确认
信息足够后，发起确认：
> "基于我们聊的内容，我整理了核心需求。在生成描述之前，还有没有要补充的？"

### 5. 输出项目描述
用户确认后，生成初始项目文档：
- 创建 `specs/` 目录
- 创建 `specs/PROJECT-CONTEXT.md`
- 创建 `project-status.json`（初始化为 `intake` 阶段）

## Output format

`project-status.json` 初始内容：
```json
{
  "project_name": "",
  "current_stage": "intake",
  "completed_stages": [],
  "active_feature": null,
  "features": [],
  "bugs": [],
  "technical_debt": [],
  "decisions": [],
  "last_updated": ""
}
```

## Quality checklist
- [ ] 用户的核心痛点、目标用户、关键功能被清晰记录
- [ ] 用户确认了项目描述的准确性
- [ ] `project-status.json` 已创建
- [ ] `specs/` 目录已创建

## Guardrails
- 不猜测用户没说过的需求——信息不足就继续问
- 不给技术建议——这是技术栈选型阶段的事
- 不输出结构化问卷——保持自然对话
- 用户说"差不多就这些"时，再做一次最终确认，不直接跳过

## Common mistakes to avoid
- 用户说"我想做个博客"就直接输出完整的产品概述文档——太急，关键维度可能没覆盖
- 帮用户脑补他没有提到的功能——"你的博客可能需要评论功能吧？"这是需求澄清阶段的事
- 纠结于细节（"每篇文章最大多少字？"）而忽略了核心方向
