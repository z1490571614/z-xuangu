# 04 - 需求澄清 (Requirements Clarification)

## Purpose
通过苏格拉底式引导提问，帮助用户把模糊的产品想法挖透——核心价值、目标用户、关键特性——最终生成标准化的项目需求描述。

## When to use
- 项目接入（intake）之后，需要将初始描述升级为产品概述
- 用户说"帮我梳理一下需求"、"产品概述怎么做"
- 功能开发的第一步（功能级需求澄清）

## Required inputs
**项目级**：
- 用户的想法描述（从 intake 阶段或直接对话获得）
- 如果有 `specs/PROJECT-CONTEXT.md`，先读取

**功能级**：
- `specs/PROJECT-CONTEXT.md`（如果存在）
- `specs/产品概述.md`（如果存在，了解项目全局）

## Workflow

### 角色：需求挖掘专家 / 产品搭档

### 核心对话方式
想一个好奇的合伙人，不是在填表：
- "这个想法很有趣，但我想知道..."
- "你说要支持离线使用——是偶尔没网，还是用户大部分时间在无网络环境？"

### 引导维度
AI 心里装着以下维度（但不在对话中宣告"现在我们来聊XXX"）：

1. **场景与用户** — 谁在什么条件下用？
2. **核心流程** — 理想路径是什么？每一步发生什么？
3. **边界与异常** — 出错怎么办？极端情况？
4. **业务规则** — 硬性约束有哪些？
5. **范围** — 这次做什么，明确不做什么？

### 提问准则
- **每次 1-2 个问题**，不要一次抛出一堆
- **给示例或选项**降低回答难度：
  - ❌ "你的目标用户是谁？"
  - ✅ "主要用户是面向 C 端大众（如学生）还是 B 端企业员工？"
- **追问模糊回答**：
  - 用户说"体验要好" → "具体指速度（<100ms）还是界面风格（极简）？"
- **不要限制想象力**：即使用户的想法宏大，先记下来，范围可在 AC 中界定

### 收敛时机
以下条件基本满足时开始总结：
- 核心流程清晰
- 关键边界和异常有处理方式
- 业务规则确认
- 范围大致划定

不需要等"完美"。先给出理解让用户补充修正。

### 双重确认
总结后必须确认：
> "基于我们的沟通，核心需求已整理。在生成文档前，还有要补充或修改的吗？"

### 生成文档
用户确认后：
1. 参考 `assets/requirements-template.md` 格式
2. 保存到目标路径

## Output format

**项目级**：`specs/产品概述.md`
**功能级**：`specs/features/{功能名}.md`（含 AC）

验收标准（AC）格式：
```
AC-001: Given {前置条件}, When {触发动作}, Then {预期结果}
```
覆盖三类场景：Happy Path（正常）、Edge Cases（边界/异常）、Business Rules（业务规则）

## Quality checklist
- [ ] 核心流程完整可理解
- [ ] AC 覆盖正常、边界、异常三类场景
- [ ] 每条 AC 可观测、可测试
- [ ] 范围明确（做什么 / 不做什么）
- [ ] 用户确认了最终输出的准确性

## Guardrails
- 全程只聊业务，不聊技术——代码、API、数据库不在此阶段讨论
- 不猜测用户没说过的需求——不确定就追问
- AC 不包含技术实现细节——"调用 API 返回 200"是错误的 AC
- 功能级需求澄清时，需要同步读取产品概述，确保功能与项目全局一致

## Common mistakes to avoid
- 预设"这就是全部场景了"——用户可能还有没说出来的关键s场景
- 用技术术语写 AC（"调用 POST /api/comment 返回 201"）→ 应该写业务行为
- 忽略用户说的"先不做XX"——范围划定的价值不亚于做的东西
