# 07 - 开发规范制定 (Dev Standards)

## Purpose
基于技术栈制定完整的开发规范——代码风格、命名约定、Git 工作流、AI 协作协议。每一条规则都具体到可以直接执行。

## When to use
- 项目结构确定后，需要制定开发规范时
- 团队有新成员加入需要统一规范时

## Required inputs
- `specs/技术栈.md` — 决定语言、框架、工具链
- `specs/项目结构.md` — 决定模块划分和依赖方向
- `specs/产品概述.md` — 了解业务领域（影响分支策略、测试复杂度）

## Workflow

### 角色：技术委员会

### 规范覆盖范围
按模板章节逐一生成：

1. **语言与运行时** — 版本、包管理器
2. **代码风格与 Lint** — ESLint/Prettier/Biome 具体配置
3. **命名约定** — 文件、目录、组件、函数、变量、类型的命名（表格形式 + 示例）
4. **语言特定规范** — 禁止 `any`、`import type`、正反对比代码
5. **框架特定规范** — 组件结构模板、导入顺序、Hooks 使用规则
6. **样式规范** — 唯一方案（Tailwind / CSS Modules）、禁止的写法
7. **Git 工作流** — 分支策略、Conventional Commits、合并策略
8. **注释规范** — 什么时候写/不写、TODO/FIXME 格式
9. **错误处理** — 服务端/客户端模式、用户提示规则
10. **测试规范** — 策略、工具、优先级、文件位置
11. **依赖管理** — 安装命令、审查原则、版本锁定
12. **环境变量** — 命名规则、文件管理、安全红线
13. **AI 协作协议** — 必读文档清单、编码规则、写完检查清单、错题本
14. **依赖方向** — 合法/禁止方向（ASCII 图）
15. **安全编码** — 输入校验、注入防护、密钥管理

### 规范质量标准
- **具体 > 抽象**：每条规则配代码示例（正确 vs 错误）
- **可执行 > 可解释**：能直接当 checklist 用
- **适配 > 通用**：所有规则针对本项目技术栈定制
- **完整 > 精简**：模板所有章节必须填写，不适用的标注原因

### 生成文档
1. 参考 `assets/dev-standards-template.md` 格式
2. 逐章节生成
3. 进行一致性检查（是否与技术栈、目录结构、项目已有约定一致）
4. 保存到 `specs/开发规范.md`

## Output format
`specs/开发规范.md`，包含 15 个章节，每条规则有代码示例。

## Quality checklist
- [ ] 所有规则针对本项目技术栈定制
- [ ] 关键规则有正反对比代码示例
- [ ] AI 协作协议具体、可执行
- [ ] 各章节之间无矛盾
- [ ] 示例代码语法正确

## Guardrails
- 不出现模糊表述（"遵循最佳实践"、"建议使用"）→ 用"必须"或"禁止"
- 不生成与项目技术栈无关的规则（如 TypeScript 项目不写 Python 规范）
- 先读全 `specs/` 下已有文档，不遗漏已定义的约束

## Common mistakes to avoid
- 复制粘贴通用规范模板，不根据项目定制
- 规则太"软"——"建议用 PascalCase"→ 明确"组件文件用 PascalCase"
- AI 协作协议写成空话——"写代码前先思考"不应该出现
