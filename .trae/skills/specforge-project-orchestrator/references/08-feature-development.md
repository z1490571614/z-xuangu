# 08 - 功能开发 (Feature Development)

## Purpose
定义功能开发从需求到编码的完整闭环流程。适用于：
- **新功能开发**：需求澄清 → 技术方案 → 任务规划 → 编码实现
- **功能迭代（evolution）**：对已完成或进行中功能的增量修改

## When to use
- 用户说"我想做一个XX功能"、"开发XX功能"
- 用户说"XX功能要改一下"、"给XX加个功能"
- 路线图上某功能的开发

## Required inputs

### 新功能开发
- `specs/PROJECT-CONTEXT.md`（如果存在）
- `specs/产品概述.md`、`specs/技术栈.md`、`specs/项目结构.md`、`specs/开发规范.md`

### 功能迭代
- `specs/features/{功能名}.md`（需求文档）— 必须存在
- `specs/features/{功能名}_技术方案.md` — 如果存在则读取
- `specs/features/{功能名}_任务规划.md` — 如果存在则读取
- 相关代码文件

## 新功能开发流程

### 阶段 1：功能需求澄清
**角色**：产品搭档
**流程**：通过自然对话挖掘需求，产出验收标准（AC）
**参考**：`references/04-requirements.md`

**核心产出**：
- `specs/features/{功能名}.md` — 含 Given-When-Then 格式的 AC
- AC 覆盖正常、边界、异常三类场景
- 明确范围（做什么/不做什么）

**质量要点**：
- 全程不聊技术——不出现 API、数据库、代码
- 每条 AC 必须可观测、可测试
- AC 是后续所有阶段的锚点——写不好，后面全返工

### 阶段 2：技术方案设计
**角色**：技术搭档（系统架构师）
**流程**：从 AC 反推技术设计，做真正的设计决策

**核心动作**：
1. **理解现有代码**：涉及哪些模块？有什么可复用？会影响什么？
2. **数据模型**：需要存什么？怎么建索引？
3. **API 设计**：路径、方法、参数、响应、错误处理
4. **核心逻辑**：从 AC 三类场景反推实现方案

**决策确认**：
- 有多种可行方案 → 列出、分析、推荐
- 需要新依赖 → 说明为什么现有工具不够
- 影响现有功能 → 明确影响范围

**核心产出**：
- `specs/features/{功能名}_技术方案.md`
- AC 覆盖总表：每条 AC 都能追溯到技术设计点

### 阶段 3：任务规划
**角色**：技术主管
**流程**：按垂直切片策略拆分任务

**垂直切片**：
- 按用户行为拆分（不是按技术层）
- ❌ 阶段：建表 → 写 API → 做 UI
- ✅ 切片："发表评论"（表+API+UI）→ "查看列表" → "回复评论"

**每个任务包含**：
- 通俗解释（零技术术语的用户可见变化）
- 验证标准（能直接作为 TDD RED 阶段测试用例）
- 依赖标注（Mermaid 图展示）
- 对应的 AC 编号和技术方案章节

**关键变化（V3）**：
- 测试不是独立任务——TDD 内嵌在每个任务中
- 验证标准是核心产出——决定了 TDD 能不能启动

**核心产出**：
- `specs/features/{功能名}_任务规划.md`

### 阶段 4：编码实现
**角色**：严格践行 TDD 的开发者
**流程**：逐阶段执行任务，每个任务走 RED → GREEN → REFACTOR

**RED**：根据验证标准写测试 → 运行 → 确认失败（原因：功能未实现）
**GREEN**：写最少代码让测试通过 → 跑全量测试防回归
**REFACTOR**：在测试保护下优化 → 简单任务可跳过

**铁律**：没有失败的测试，就不写实现代码

**验收测试**：
- UI 变化的任务：Playwright 浏览器验证
- 纯后端任务：单元测试覆盖
- 数据库变更：直接查询验证

**核心产出**：
- 代码 + 测试
- 更新后的任务规划（已完成任务打勾）
- `docs/开发记录/{功能名}_阶段N_完成报告.md`

## 功能迭代（Evolution）流程

### 触发条件
- 功能已开发完成（或开发中），用户想要增量修改
- **不是重做**——如果变更超过原功能 70%，建议重走新功能流程

### 变更类型
| 类型 | 描述 | 示例 |
|------|------|------|
| 微调 | 小改动，不涉及新 AC | 调整排序方式 |
| 扩展 | 加新东西，新增 AC | 加搜索历史 |
| 重构 | 动核心结构 | LIKE → Elasticsearch |

### 工作流程
**角色**：产品架构师

1. **理解变更意图**：对话搞清楚要改什么
2. **影响分析**（四层面）：
   - 需求：AC 增删改
   - 技术：数据库、API、组件
   - 代码：受影响文件、测试
   - 任务：进行中的任务是否被波及
3. **增量更新文档**：就地修改 + 末尾追加变更日志
4. **生成增量任务计划**：生成 `specs/features/{功能名}_变更任务_{CR序号}.md`

**核心原则**：改，不是重做。没受影响的内容原封不动。

## Quality checklist
- [ ] 需求阶段产出完整的 AC（三类场景齐全）
- [ ] 技术方案覆盖每条 AC（可追溯）
- [ ] 任务规划使用垂直切片策略
- [ ] 验证标准能直接转为测试用例
- [ ] 编码阶段每个任务走 TDD 循环
- [ ] 功能迭代的增量文档只改受影响部分

## Guardrails
- 需求阶段不讨论技术实现
- TDD 铁律不可违反
- 任务未完成不跳过
- 功能迭代不改无关内容

## Common mistakes to avoid
- 需求没想清楚就写代码 → 返工率极高
- 按技术层水平切片 → AI 上下文丢失，无法端到端验证
- 验证标准写得太模糊 → TDD 启动不了
- 跳过 REFACTOR → 代码质量持续下降
