# 11 - 文档同步 (Doc Sync)

## Purpose
保持代码和文档的一致性。代码改了，文档没更新是常见问题，这个流程帮用户主动发现并修复不一致。

## When to use
- 用户说"文档跟代码对不上了"、"同步一下文档"
- 功能开发过程中做了计划外的改动
- 编码完成后检查文档是否需要更新
- 定期维护时

## Required inputs
- `specs/` 下的相关文档
- 对应的代码文件
- `project-status.json`

## Workflow

### 角色：文档维护者

### 同步场景

**场景 1：代码改了，文档没跟上**
- 用户开发中临时加了功能、改了逻辑
- 对比代码现状和已有文档
- 增量更新文档（不重写）

**场景 2：文档改了，代码没跟上**  
- 通过 feature-evolution 修改了需求/技术方案
- 对比差异生成代码变更清单

**场景 3：全局审计**
- 用户要求全面检查文档和代码一致性
- 扫描 `specs/` 和代码目录
- 列出所有不一致项，逐一确认

### 同步方式
- **增量为主**：就地修改受影响的章节
- **不做全量重写**：除非文档与代码差异超过 70%
- **记录变更**：文档末尾追加变更日志

### 同步检查清单
- 需求文档中的 AC 是否与实际功能一致？
- 技术方案中的 API 是否与代码一致？
- 数据库结构是否与文档描述一致？
- 新增文件是否更新了项目结构文档？

## Output format
- 更新的文档（增量修改 + 变更日志）
- 同步报告（哪些文档更新了，哪些不需要）
- 遗留不一致清单（确认暂时不需要同步的差异）

## Quality checklist
- [ ] 扫描了所有可能受影响的文档
- [ ] 只修改了实际有差异的部分
- [ ] 变更日志记录了改动
- [ ] 不确定的差异问了用户再改

## Guardrails
- 不改代码——文档同步只更新文档。如果发现代码需要改，建议走 Bug 修复或功能迭代
- 不确定代码行为是否正确时，先确认再改文档
- 不做全量重写——文档质量可能下降

## Common mistakes to avoid
- 看到差异就直接修改文档——需要先确认：是代码对了文档错了，还是反过来的？
- 只同步需求文档，不管技术方案和任务规划
- 全量重写导致已有的细节信息丢失
