# 00 - 意图路由 (Intent Routing)

## Purpose
支持自然语言意图识别。用户不需要使用命令，像跟开发搭档聊天一样表达需求。你根据语义理解加载对应的 workflow module。

## When to use
每次用户发起对话时自动使用。如果已有活跃的任务上下文（如正在开发某功能的第 3 阶段），直接进入对应流程，不需要重新路由。

## Required inputs
- 用户发出的自然语言消息
- `project-status.json`（如果有，用来判断当前项目阶段）

## Workflow

1. **读取上下文**：检查 `project-status.json`，了解当前阶段和活跃任务
2. **分析意图**：理解用户消息的语义，识别意图类型
3. **路由分发**：加载对应的 workflow module
4. **歧义处理**：无法确定意图时，用自然对话向用户澄清

## Intent mapping

| 用户说的话（示例） | 意图 | 加载模块 |
|---|---|---|
| "我想做一个XX" / "帮我起个项目" | 新项目启动 | `03-project-intake.md` |
| "项目进度怎样了" / "下一步做什么" | 查看进度 | `12-state-management.md` |
| "我想给XX加个功能" / "新功能" | 新功能开发 | `08-feature-development.md` |
| "XX功能想改一下" / "加个搜索历史" | 功能迭代 | `08-feature-development.md`（evolution 流程） |
| "XX出bug了" / "XX不工作" | Bug 修复 | `09-bug-fix.md` |
| "用什么数据库好" / "这个技术选型" | 技术咨询 | `05-tech-stack.md` |
| "文档跟代码对不上了" / "同步一下文档" | 文档同步 | `11-doc-sync.md` |
| "帮我看看这段代码" / "代码审查" | 代码审查 | `10-code-review.md` |
| "生成 Trae rules" / "生成项目规则" | Trae 适配 | `14-trae-adapter.md` |
| "帮我补点测试" / "这个模块没测试" | 测试生成 | `08-feature-development.md`（TDD 流程） |
| "重构一下XX" / "代码太乱" | 重构 | `10-code-review.md` |

## Output format
不输出结构化格式。根据路由结果，直接进入对应 workflow module 的角色设定和流程。用户只需要感受到一个连贯的对话体验。

## Quality checklist
- [ ] 意图识别准确——用户在问技术问题时不跑去需求澄清
- [ ] 不影响已有任务上下文——如果正在执行功能的第 2 阶段，不要因为用户随口提了"下次加个 XX"就切断当前流程
- [ ] 歧义时优先澄清，不猜测后默默执行

## Guardrails
- 不要求用户记忆或使用命令触发词
- 不把全部意图列表打印给用户让他选
- 意图识别失败时，用自然对话引导，不报错
- 如果项目没有任何上下文（`project-status.json` 不存在且 `specs/` 为空），默认进入"新项目启动"流程

## Common mistakes to avoid
- 用户说"看看 XX 功能的代码"，把它当代码审查而不是功能开发
- 用户说"这个逻辑需要调整"，先判断是 bug（行为与设计不符）还是需求变更（行为符合设计但要改需求）
- 跳过多重意图判断——用户一句话可能包含"先把 A 功能做完，然后帮我看看 B 的测试"，需要优先处理 A
