# 05 - 技术栈选型 (Tech Stack)

## Purpose
基于产品需求，推荐最合适的技术栈（而不是最热门的），并生成标准化技术栈文档。

## When to use
- 产品概述确定后，需要选择技术栈时
- 用户问"用什么框架/数据库好"
- 用户在多个技术方案之间纠结

## Required inputs
- `specs/产品概述.md` — 了解项目规模、需求、约束
- `specs/PROJECT-CONTEXT.md`（如果存在）
- 用户的技术偏好（如有）

## Workflow

### 角色：CTO / 技术架构师

### 需求分析
从产品概述中提取关键约束：
- 项目规模（小中大）
- 用户量级
- 实时性要求
- 是否需要 SEO
- 开发周期和团队规模
- 成本限制

### 推荐策略
- **主动推荐**：不等待用户逐项选择。根据分析给出 1-2 套完整组合方案
- **首选成熟方案**：用户无偏好时，推荐最成熟、生态最好的方案
- **适配 AI 开发**：优先选 AI 训练数据多、社区资源丰富的技术——AI 写起来更顺手

### 需要确认时
当有多种合理方案且差异显著时，提供选项：
- 列出方案 A / B / C
- 分析各方案利弊
- 给出推荐和理由

**示例**：
> "数据存储有两个方向：PostgreSQL（功能最强、适合结构化业务）和 MongoDB（灵活、开发快）。考虑到你的业务模型比较固定，建议选 PostgreSQL。"

### 生成文档
1. 参考 `assets/tech-stack-template.md` 格式
2. 包含：前端、后端、数据库、部署、工具链
3. 每个选择附带理由
4. 保存到 `specs/技术栈.md`

## Output format
`specs/技术栈.md`，包含：
- 整体架构图
- 各技术层选型及理由
- 依赖清单（核心库及版本）
- 未选方案的说明（为什么不用）

## Quality checklist
- [ ] 每个选择都有理由，而不是"因为流行"
- [ ] 方案完整——从前端到部署全覆盖
- [ ] 考虑了项目约束（成本、人才、时间）
- [ ] 用户确认了最终方案

## Guardrails
- 不推荐用户用不了的技术（如要求精通 Rust 但用户是前端背景）
- 不推荐刚发布、社区小的新框架（AI 训练数据少、踩坑多）
- 技术栈选定后不与现有 `specs/技术栈.md` 冲突（除非用户要求重新选型）
- 不替用户决策需要成本投入的方案（如付费云服务 vs 自建）

## Common mistakes to avoid
- 机械套用上一项目技术栈，不分析当前项目特点
- 追求"技术先进性"而忽视实际约束
- 给太多选项让用户决策疲劳——2-3 个足够
