# 选股策略回测有效性验收 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 建立一套“用回测策略生成历史候选池，并用最近真实生产选股列表验收策略有效性”的开发方案，作为 `leader_main_t0_lgbm` 训练样本来源的可信度检查。

**Architecture:** 复用现有 `leader_main_t0_training_sample` 作为回测候选池，复用 `selection_record + selected_stock` 作为真实生产对照列表，新增独立服务计算逐日相似度、缺数据状态和整体验收结论。先以服务层和脚本落地，随后挂接到 `/api/v1/backtest`，最后作为训练任务的前置可选检查。

**Tech Stack:** FastAPI, SQLAlchemy, SQLite WAL, pytest, 现有 `backend.services.backtest` 与 `backend.services.model_engine`。

## 1. 背景

当前系统已经具备完整的竞价回测与 LightGBM 训练管线：

| 模块 | 当前能力 |
|------|----------|
| `selection_record` | 保存真实生产选股批次 |
| `selected_stock` | 保存真实生产选股明细 |
| `stock_feature_snapshot` | 保存生产选股后的特征快照 |
| `stock_auction_open` | 保存历史集合竞价数据 |
| `leader_main_t0_training_sample` | 保存回测候选样本与 T+0 标签 |
| `LeaderMainT0FeatureBuilder` | 按策略过滤生成回测候选样本 |
| `LeaderMainT0LabelBuilder` | 生成非一字触板且收板标签 |
| `lightgbm_service` | 训练和预测 `leader_main_t0_lgbm` |

用户当前遇到的问题是：真实生产环境中每日选股列表只有少量日期，不能直接作为大规模训练集。解决方向是用现有选股策略做历史回测生成训练样本，再用最近真实生产选股列表校验该回测策略是否足够贴近生产策略。

本开发文档的目标不是重新造一套选股系统，而是在现有管线之上增加“策略有效性验收”能力。

## 3. 验收口径

### 3.1 推荐口径

策略有效的默认判定：

```text
参与验收交易日的平均召回率 >= 70%
```

召回率定义：

```text
recall = overlap_count / production_count
```

其中：

- `production_count`：某交易日真实生产选股数量。
- `backtest_count`：同交易日回测候选样本数量。
- `overlap_count`：真实生产列表与回测候选列表的交集数量。

### 3.2 缺数据日处理

当某交易日满足以下条件时，标记为 `insufficient_backtest_data`，默认不纳入通过/失败判断：

```text
production_count > 0
and backtest_count == 0
and exclude_insufficient_data = true
```

默认排除的原因是：该日期无法区分“策略未命中”和“回测样本未构建/竞价数据缺失/日线缓存缺失”。

### 3.3 辅助指标

Jaccard 用于观察“回测多选”的程度：

```text
jaccard = overlap_count / union_count
```

Jaccard 不作为第一期硬性通过标准。原因是训练阶段允许回测候选池比生产列表略宽，以增加样本覆盖面；但回测列表过宽会降低训练样本纯度，所以必须展示。

## 4. 设计原则

1. 优先复用现有数据源。

   真实列表使用 `selection_record + selected_stock`；回测列表使用 `leader_main_t0_training_sample`；不新增外部 API；不重新直连 Tushare 获取新闻或龙虎榜等无关数据。

2. 不污染训练样本。

   验收服务只读取数据，不在 `leader_main_t0_training_sample` 中写入验收结果。训练样本表继续只承载候选特征和标签。

3. 缺数据显式降级。

   无回测样本、无真实列表、真实列表为空都要给出明确 `status`，不把缺数据误判为策略失败。

4. 服务层先行。

   第一阶段以纯服务函数和单元测试完成核心计算，再接 API 和脚本，最后挂到模型训练任务。

5. 可追溯。

   每个交易日输出真实列表、回测列表、重合列表、漏选列表、额外回测列表和过滤状态，方便人工复核。

## 7. 服务设计

### 7.1 数据类

文件：`backend/services/backtest/selection_strategy_validation_service.py`

```python
from dataclasses import dataclass
from typing import List


@dataclass
class DailySelectionValidation:
    trade_date: str
    status: str
    production_count: int
    backtest_count: int
    overlap_count: int
    union_count: int
    recall: float | None
    jaccard: float | None
    production_codes: List[str]
    backtest_codes: List[str]
    overlap_codes: List[str]
    missing_codes: List[str]
    extra_backtest_codes: List[str]


@dataclass
class SelectionStrategyValidationResult:
    start_date: str
    end_date: str
    min_avg_recall: float
    exclude_insufficient_data: bool
    status: str
    validated_days: int
    excluded_days: int
    avg_recall: float | None
    avg_jaccard: float | None
    total_production_count: int
    total_backtest_count: int
    total_overlap_count: int
    days: List[DailySelectionValidation]
```

### 7.2 服务类

```python
class SelectionStrategyValidationService:
    def __init__(self, session_factory=SessionLocal):
        self.session_factory = session_factory

    def validate(
        self,
        start_date: str,
        end_date: str,
        min_avg_recall: float = 0.70,
        exclude_insufficient_data: bool = True,
    ) -> SelectionStrategyValidationResult:
        ...
```

### 7.3 状态定义

| 状态 | 含义 | 是否参与均值 |
|------|------|--------------|
| `validated` | 真实列表和回测列表都存在，可计算指标 | 是 |
| `insufficient_backtest_data` | 有真实列表但无回测列表 | 默认否 |
| `no_production_data` | 无真实生产列表 | 否 |
| `empty_production_list` | 真实记录存在但股票数为 0 | 否 |

整体 `status`：

| 状态 | 条件 |
|------|------|
| `passed` | `validated_days > 0` 且 `avg_recall >= min_avg_recall` |
| `failed` | `validated_days > 0` 且 `avg_recall < min_avg_recall` |
| `insufficient_data` | 没有任何可参与验收的交易日 |

```json

}
```

服务异常返回 `500`，错误信息要包含“策略回测有效性验收失败”，便于定位。

## 11. 测试矩阵

| 场景 | 期望 |
|------|------|
| 回测列表完全覆盖真实列表 | `status=passed` |
| 平均召回率低于阈值 | `status=failed` |

| 脚本验收失败 | 退出码 `1` |

## 13. 风险与边界

### 13.1 真实生产列表少

当前只有 5 个真实生产选股日，统计置信度不高。因此验收结论应表述为：

```text
当前回测策略在最近可对照生产日期上通过相似度验收
```

不能表述为：

```text
该策略长期稳定有效
```

### 13.2 回测候选池过宽

如果召回率高但 Jaccard 低，说明回测多选了大量生产不会选的股票。第一期不直接判失败，但要在输出中突出 `extra_backtest_codes`，后续可增加：

```text
avg_jaccard >= 0.50
```

作为辅助门槛。

### 13.3 生产快照不能用于自证

`stock_feature_snapshot` 来自生产选股后沉淀，若直接作为训练样本源，会天然贴近真实列表。验收时应优先对比 `leader_main_t0_training_sample`，并在输出里标明样本来源。

### 13.4 缺数据日不能静默忽略

`20260515` 这类日期必须展示为 `insufficient_backtest_data`，否则用户无法判断是策略问题还是数据链路问题。

## 15. 后续扩展

第二期可以把验收结果接入模型训练任务：

- 测试训练只展示验收摘要，不阻止训练。
- 正式训练可配置 `require_selection_validation=true`。
- 验收失败时不激活模型版本。
- 前端 `ModelCenter.vue` 展示最近验收结果、缺数据日期和漏选股票。

第三期可以把验收结果持久化为审计记录：

- 新增 `model_strategy_validation` 表。
- 保存每次训练前的验收快照。
- 将模型版本、训练任务、策略验收结果关联起来。
